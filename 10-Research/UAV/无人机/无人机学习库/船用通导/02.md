# 从设备名走向接口表

## 上一课反馈处理

- 纠错要点：你已经抓到“传感器采集、显示、决策、报警”等闭环环节，但还缺少两个关键动作：把数据从哪里来、到哪里去说清楚；把失效后果说清楚。
- 标准解/更优解：更完整的闭环可以写成“传感器采集 -> 数据分发 -> 显示/融合 -> 人或自动系统决策 -> 报警/记录 -> 复盘”。设备分析建议用“位置、输入、输出、失效后果”四格表。
- 是否推进：补课。
- 一句话理由：你理解了系统不是设备堆砌，但还没有稳定掌握通导集成最核心的接口思维。

## 目标（预计 5-10 分钟）

本课只练一个能力：看到一台通导设备，不再只说“它是干什么的”，而是能写出：

- 它在系统闭环中的位置。
- 它需要哪些输入。
- 它向谁输出什么。
- 它失效后会影响哪些系统能力。

这就是从“设备清单思维”进入“系统集成思维”的第一步。

## 核心问题

为什么通导学习必须从接口表开始？

因为船上验收关心的不是“设备名是否完整”，而是每条关键数据链是否真的通了。例如 DGPS、AIS、ECDIS、VDR 单机都可能正常，但如果 DGPS 没有把位置数据正确送到 AIS，AIS 发出的本船动态信息就可能不完整；如果雷达或测深仪没有被 VDR 正确记录，事故复盘能力就不完整。

所以，系统能力藏在箭头里：

```text
设备 A --某种数据/接口--> 设备 B
```

会画箭头，才开始会做系统。

## 第一性原理/核心模型

通导设备的系统价值可以用一个四格模型判断：

```text
位置：它属于采集、分发、显示、决策、报警、记录中的哪一环？
输入：它依赖谁给它数据、供电、信号或控制？
输出：它把什么数据送给谁？
失效后果：它坏了、数据错了、接口断了，会让哪种航行能力下降？
```

注意：失效不只等于设备黑屏。通导系统里常见的失效有三类：

1. 单机失效：设备本身不能工作。
2. 数据失效：设备显示正常，但输入数据错误、丢失或延迟。
3. 接口失效：两台设备都正常，但数据没有正确传过去。

例如 AIS 主机能开机，不代表 AIS 系统一定合格。它还需要正确接收本船位置、航向、航速等数据，并把目标信息输出到雷达、ECDIS、VDR 等设备。

## 关键概念与例子

下面用 H2623 导航系统设备协议里的常见设备做四格拆解。

| 设备   | 位置         | 输入                              | 输出                                | 失效后果                               |
| ---- | ---------- | ------------------------------- | --------------------------------- | ---------------------------------- |
| DGPS | 传感器采集/定位授时 | GNSS 天线信号、电源                    | 位置、时间、对地速度等给雷达、AIS、ECDIS、VDR、自动舵等 | 本船位置源不可靠，ECDIS、AIS、VDR、自动舵等后续能力受影响 |
| 电罗经  | 传感器采集/航向基准 | 电源、内部陀螺/传感单元                    | 船首向给雷达、AIS、ECDIS、自动舵、VDR 等        | 目标方位、航迹控制、记录数据可能错误                 |
| 计程仪  | 传感器采集/航速   | 换能器或传感器、电源，可能接收 DGPS/罗经辅助数据     | 航速、航程给雷达、AIS、ECDIS、VDR、推进系统等      | 避碰计算、航迹推算、记录数据受影响                  |
| 测深仪  | 传感器采集/水深   | 换能器、电源                          | 水深、浅水报警给 ECDIS、VDR、Conning 等      | 浅水航行风险上升，VDR 缺少关键事故数据              |
| AIS  | 目标协同/数据交换  | DGPS、电罗经、计程仪、本船静态信息、电源          | 本船信息发给外部船舶；周边 AIS 目标给雷达、ECDIS、VDR | 协作目标识别能力下降，但不能替代雷达                 |
| VDR  | 记录/复盘      | 雷达、VHF、测深仪、DGPS、电罗经、AIS、报警等多源数据 | 事故后回放数据、保护存储                      | 事故调查和责任追溯能力下降                      |

把这张表看懂后，再看技术协议里的“输入信号”“输出信号”“信号分配器”“电源接口”“供货范围”，就不会觉得它们是杂项。它们是在定义系统能力能不能闭环。

一个更小的例子：

```text
DGPS -> AIS -> ECDIS/VDR
```

这条链路里至少有三个检查点：

- DGPS 是否输出正确位置和时间。
- AIS 是否接收到本船动态数据并正确发送。
- ECDIS/VDR 是否接收到 AIS 目标或相关记录数据。

如果只检查 AIS 屏幕亮了，就漏掉了系统级验收的重点。

## 常见坑

1. 把输入输出写成设备功能

“DGPS 是定位设备”不是输入输出。更好的写法是：“输入 GNSS 天线信号，输出位置、时间、对地速度给 ECDIS、AIS、VDR 等。”

2. 把失效后果写得太泛

“坏了会影响航行”太泛。更好的写法是：“DGPS 数据异常会影响 ECDIS 本船位置显示、AIS 本船动态信息、VDR 记录完整性和自动舵航迹控制。”

3. 以为证书能替代集成测试

CCS 产品证书证明设备本身符合相应要求，但不能证明它和其他设备之间的数据链路已经正确接通。系统验收还要看安装、接口、报警、记录、显示、系泊试验、航行试验。

4. 忽略电源与天线

通导设备常见主电源是 AC220V，同时可能有 DC24V 备用或转换。天线位置、电缆长度、接地、电磁兼容也会直接影响系统可用性。
